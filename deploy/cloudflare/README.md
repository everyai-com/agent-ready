# agent-ready — Cloudflare Worker gateway

Deploys your [agent-ready](https://github.com/everyai-com/agent-ready) MCP gateway as a
Cloudflare Worker in your own Cloudflare account. Your Supabase service key stays in
Cloudflare's secret store — it never touches agent-ready's infrastructure.

[![Deploy to Cloudflare](https://deploy.workers.cloudflare.com/button)](https://deploy.workers.cloudflare.com/?url=https://github.com/everyai-com/agent-ready/tree/main/deploy/cloudflare)

## What this deploys

- `GET /` — a landing page confirming the gateway is live and how to connect to it.
- `GET /health` — a JSON health check.
- `POST /mcp` — the MCP server (Streamable HTTP / JSON-RPC 2.0), generated from
  `manifest.json` and backed by your Supabase project via `SupabaseAdapter`.

## Deploy

### 1. One-click

Click **Deploy to Cloudflare** above. Cloudflare will fork this template into your own
repo and Worker, then prompt you for the secrets below.

### 2. Manual

```bash
cd deploy/cloudflare
npm install
```

Replace `manifest.json` with the capability manifest generated by the agent-ready
wizard for your project (or hand-edit the sample — see the schema in
`@agent-ready/core`).

Set your secrets — these are stored encrypted in Cloudflare, not in this repo:

```bash
npx wrangler secret put SUPABASE_URL
npx wrangler secret put SUPABASE_SERVICE_KEY

# Optional: require a bearer token on /mcp. Comma-separate multiple keys to
# issue distinct tokens per agent.
npx wrangler secret put GATEWAY_API_KEY
```

Deploy:

```bash
npx wrangler deploy
```

Wrangler prints your Worker's URL, e.g. `https://agent-ready-gateway.<you>.workers.dev`.
Your MCP endpoint is `<that URL>/mcp`.

### Local development

```bash
npx wrangler dev
```

Wrangler reads secrets for local dev from `.dev.vars` (gitignored). Create one:

```
SUPABASE_URL=https://xxxx.supabase.co
SUPABASE_SERVICE_KEY=eyJ...
GATEWAY_API_KEY=dev-key-123
```

## Connecting an agent

Point any MCP-compatible client (Claude, Claude Code, etc.) at `<your-worker-url>/mcp`.
If `GATEWAY_API_KEY` is set, send it as `Authorization: Bearer <key>`.

## Safety notes

- Only capabilities marked `enabled: true` in `manifest.json` are ever exposed as
  tools — nothing is introspected-and-exposed automatically.
- Writes, if you enable any, run through the same manifest-declared, schema-validated
  path as reads. There is no raw-SQL or arbitrary-filter escape hatch in the generated
  gateway.
- Rotate `GATEWAY_API_KEY` any time with `wrangler secret put GATEWAY_API_KEY` followed
  by a redeploy; omit it entirely to require no auth (not recommended in production).
